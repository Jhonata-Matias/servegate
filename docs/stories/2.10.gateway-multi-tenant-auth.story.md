## Story

**As a** owner do servegate operando o gateway em fase alpha com múltiplos consumidores externos,
**I want** que o `validateAuth` do gateway aceite um conjunto de `GATEWAY_API_KEY`s distintas (uma por tenant), com cada request mantendo o `api_key_hash` já emitido em logs/quota,
**so that** eu possa emitir/rotacionar/revogar credencial por tenant sem coupling operacional, isolar quota de vídeo automaticamente (bucket per-hash já implementado em Story 5.2) e preservar atribuição por-tenant em incident review — **sem** ainda mover para lookup em KV (adiado para Story futura quando ≥3 tenants ativos).

## Acceptance Criteria

1. **Multi-key aceito:** `validateAuth(request, keys)` recebe um `readonly string[]` de keys válidas (não mais uma única string). Retorna `null` (sucesso) se o header `X-API-Key` bate contra **qualquer** entrada da lista via comparação de tempo constante; retorna 401 caso contrário. Zero regressão no single-key path (lista de 1 elemento = comportamento atual).
2. **Comparação constant-time preservada:** para cada key candidata, `constantTimeEqual(provided, candidate)` executa sempre (não fazer short-circuit no primeiro match) — evita timing side-channel que revelaria qual key bateu.
3. **Env schema estendido:** `types.ts` `Env` interface adiciona campos opcionais `GATEWAY_API_KEY_2..GATEWAY_API_KEY_N` (mínimo suportado: N=4, alinhado ao teto pragmático da Opção 2). `GATEWAY_API_KEY` continua **required** (retrocompat). Uma helper `collectApiKeys(env: Env): string[]` filtra undefined/empty e retorna array pronto para `validateAuth`.
4. **Rotação por-tenant documentada:** `docs/usage/gateway-deploy.md` seção "Rotate gateway API key" ganha subseção "Per-tenant rotation" com fluxo: `wrangler secret put GATEWAY_API_KEY_<N>` (novo valor) → smoke test tenant afetado → `wrangler secret delete GATEWAY_API_KEY_<N>` do valor antigo. Tenant 1 (`GATEWAY_API_KEY`) permanece com fluxo existente.
5. **Revoke por-tenant:** `docs/usage/gateway-deploy.md` documenta `wrangler secret delete GATEWAY_API_KEY_<N>` como mecanismo de revoke instantâneo — próximo request do tenant revogado recebe 401. Deve incluir aviso: "quota bucket em `VIDEOS_KV` sob hash antigo permanece até TTL 48h — normal, não requer cleanup manual".
6. **Onboarding docs atualizados:** `docs/usage/dev-onboarding.md` + `.pt-BR.md` mencionam explicitamente que cada consumer recebe key distinta (não compartilhar). Fluxo de request via issue "Alpha Access Request" mantém, mas owner injeta em slot `GATEWAY_API_KEY_<próximo N livre>` em vez de reusar `GATEWAY_API_KEY`.
7. **Logs preservam `api_key_hash`:** requests de todos os tenants continuam emitindo `api_key_hash: sha256(providedKey)` em `log.ts` e nos jobs de vídeo (`video.ts:133`). Verificação: 2 tenants disparam 1 request cada; `wrangler tail` mostra 2 hashes distintos.
8. **Vitest coverage:** `gateway/tests/auth.test.ts` ganha ≥4 casos novos: (a) key da posição 1 bate, (b) key da posição N bate, (c) key inexistente rejeita, (d) header ausente rejeita mesmo com lista não-vazia. Ambiente de teste popula `GATEWAY_API_KEY` + `GATEWAY_API_KEY_2` em `env` mock.
9. **Smoke test manual (@qa):** com dois secrets configurados no Worker de produção, `curl -H "X-API-Key: <tenant-1-key>" ...` retorna 200; `curl -H "X-API-Key: <tenant-2-key>" ...` retorna 200; `curl -H "X-API-Key: <chave-inválida>" ...` retorna 401. Registrar hashes em `docs/qa/story-2.10-smoke.md` (sanitizado: só primeiros 8 chars de cada hash).
10. **Non-goal explícito:** esta story **NÃO** implementa per-tenant image quota — `RATE_LIMIT_KV` key `count:YYYY-MM-DD` permanece global (herança de Story 2.5). Documentar em Dev Notes que quando ≥3 tenants ativos, avaliar Story 2.11 (KV-backed allowlist + tenant_id lookup) que naturalmente resolve isso via `count:${date}:${tenant_id}`.

### Cloudflare Workers Secrets — Design decisions

- **Por que enumerated secrets (`_2`, `_3`, `_4`) vs comma-separated `GATEWAY_API_KEYS`?**
  - Separação por secret facilita revoke atômico (`wrangler secret delete`) por-tenant
  - Cloudflare Workers Secrets é o mecanismo canônico; parsing de CSV in-Worker adiciona superfície de bug (whitespace, encoding)
  - Trade-off: teto de 4 keys sem alterar código; > 4 já indica hora de migrar para Opção 3 (KV lookup)
- **Por que teto de 4?**
  - Alpha access está em fase controlada; 3-4 tenants é o limite pragmático antes de precisar de UI de gerenciamento
  - Se 5+ demanda chegar, Story 2.11 vira prioridade — não estender esta

### Constant-time compliance (crítico)

```ts
// CORRETO (constant-time):
let matched = false;
for (const key of keys) {
  matched ||= constantTimeEqual(provided, key);
}
if (!matched) return unauthorized();

// ERRADO (early-exit vaza qual key bateu via timing):
for (const key of keys) {
  if (constantTimeEqual(provided, key)) return null;
}
```

O padrão `||=` sem `break` garante que loop sempre executa `keys.length` iterações independente de match.

### Schema — Env vars

Antes:
```ts
GATEWAY_API_KEY: string;
```

Depois:
```ts
GATEWAY_API_KEY: string;              // required (tenant 1)
GATEWAY_API_KEY_2?: string;           // optional (tenant 2)
GATEWAY_API_KEY_3?: string;           // optional (tenant 3)
GATEWAY_API_KEY_4?: string;           // optional (tenant 4)
```

Retrocompat: consumers pré-story só precisam de `GATEWAY_API_KEY`; deploy sem `_2..N` funciona identicamente.

### IDS Check (REUSE > ADAPT > CREATE)

- **REUSE:** `constantTimeEqual` (`auth.ts:11`) — reused 1:1
- **ADAPT:** `validateAuth` — assinatura muda de single string → array; corpo interno ganha loop (~15% mudança, < 30% adaptation limit)
- **CREATE:** `collectApiKeys` — novo helper puro, sem análogo existente. Justificativa: consolida parsing em um lugar, testável isoladamente
- **Não reusável de outra pasta:** SDK client (`sdk/src/client.ts:433`) só envia key (não valida), não há análogo lá

### Non-goals (fora do escopo — parked)

1. **Per-tenant image quota** — `RATE_LIMIT_KV` continua `count:YYYY-MM-DD` global. Mudar isto requer nova key format (`count:date:tenant_id`) e mapping key → tenant_id, que só faz sentido junto de Opção 3. Ver FU-4.3.1.
2. **Tenant_id em logs** — apenas `api_key_hash` continua sendo emitido. Mapping hash → tenant_id fica no owner's password manager fora do repo (sanitização public-mirror).
3. **KV-backed allowlist com revoke em runtime** — Opção 3. Nova story quando ≥3 tenants ativos ou ≥1 tenant enterprise-tier requerer audit trail formal.
4. **Rate limit por-tenant (tokens/day)** — FU-4.3.1 já registra. Não abrir nesta story.
