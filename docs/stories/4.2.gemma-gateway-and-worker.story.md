## User Story

**As** a servegate alpha consumer (developer, SDK user, curl tester),
**I want** a `POST /v1/generate` endpoint that returns streaming SSE of Gemma 3 4B text completions through the same authenticated, rate-limited gateway that already serves image generation,
**so that** I can use one base URL + one API key for both image and text modalities, with ChatGPT-grade streaming UX and a token-budget rate-limit that fairly reflects text-gen cost.

---

## Context

Story 4.1 closed the Foundation gate (`FOUNDATION_SET` GO 5/5). ADR-0004 is **Accepted**. This story implements the two blocking Tier-1 phases of the `wf-gated-model-serve` blueprint simultaneously, because the HTTP contract (Phase 1.1) is already specified in [`gemma-gateway-decision.md`](../architecture/gemma-gateway-decision.md) §4 and unblocks backend implementation (Phase 2) in parallel.

**Phase 1 scope (Gateway):** add `POST /v1/generate` route to the existing Cloudflare Worker at `gateway/` with native SSE streaming pass-through, token-budget rate-limit, CORS preflight, client-abort → upstream-abort propagation.

**Phase 2 scope (Backend):** provision NEW dedicated RunPod Serverless endpoint `endpoint-text-gen-alpha` running `runpod-workers/worker-vllm:v2.14.0` with `google/gemma-3-4b-it` on L4 24GB flex, HF_TOKEN secret, 50GB network volume pre-warmed.

**Explicit out of scope:**
- SDK changes (Story 4.3)
- Example notebooks / CLI (Story 4.3)
- TERMS/PRIVACY Prohibited Use Policy bullets (Story 4.4)
- Premium tier `gemma-3-27b-it` on A100 (Epic 5 candidate)
- Per-key multi-tier rate limits (Story 4.3+)
- Real-time per-token mid-stream rate-limit enforcement (deferred per ADR-0004)

**Critical first task:** a BLOCKING spike validating `new Response(upstream.body, …)` SSE pass-through against a real RunPod vLLM endpoint must succeed **on Day 1** before any other Phase 1 code is written. If the spike fails, Story 4.2 pauses for re-planning (TransformStream shim option).

**Non-negotiable regression guarantee (CON-text-routing):** image-gen routes (`POST /jobs`, `GET /jobs/{id}`) MUST remain byte-identical. This story does NOT edit `gateway/src/runpod.ts`, `gateway/src/storage.ts`, `handleSubmit`, `handleStatus`, `submitJob`, or `getStatus` in `gateway/src/index.ts`. New code lives in new modules; shared helpers from `auth.ts` + new additive functions in `rate-limit.ts` are the only points of reuse.

## Dev Notes

### Isolation rule (CON-text-routing) — non-negotiable

This story MUST NOT modify these existing paths:
- `gateway/src/runpod.ts` (image-gen submit/poll)
- `gateway/src/storage.ts` (image-gen job mapping)
- `handleSubmit`, `handleStatus`, `submitJob`, `getStatus` function bodies in `gateway/src/index.ts`
- Existing functions in `rate-limit.ts` (`checkAndIncrement`, `checkAndRead`, `DAILY_LIMIT`)
- `serverless/handler.py` (image-gen)
- Existing test files (`runpod.test.ts`, `storage.test.ts`, `rate-limit.test.ts`, `integration.test.ts`, `nfr-5-kv-reads.test.ts`, `auth.test.ts`)

Additive modifications ARE allowed on:
- `rate-limit.ts` (new exports parallel to existing)
- `types.ts` (new interfaces + Env fields + LogEventName cases)
- `index.ts` (new route branches appended; existing branches byte-identical)
- `wrangler.toml` (new bindings appended)

CodeRabbit pre-PR review MUST scope-check the diff and flag any regression-risk edits.

### Why dual-gate in one story

The blueprint specifies Phase 1 and Phase 2 as separate gates (`GATEWAY_LIVE`, `BACKEND_LIVE`) and allows Phase 2 to run in parallel with Phase 1 after 1.1 contract freeze. In this story, contract is already frozen in `gemma-gateway-decision.md` — so Phase 1 implementation and Phase 2 endpoint provisioning can genuinely run concurrently. The dual-gate structure keeps workflow accounting honest without slicing the work into two stories artificially.

### Day 1 spike rule (R1 from gateway-decision §9)

The single biggest risk in Story 4.2 is SSE pass-through failing in the CF Workers runtime under load. The spike (Task 3) MUST be the first implementation activity. If it fails, Story 4.2 halts and a separate story for TransformStream shim (likely +100 LOC and +1 dev-day) is created. Do NOT proceed to Tasks 4-7 if spike fails.

### Token-budget accounting trade-off

Per `gemma-gateway-decision.md` §5 + ADR-0004: we accept KV eventual-consistency tradeoff (same tradeoff as Epic 2 R7 rate-limit). Expected overshoot under concurrent race ≤ 10%. Monitor weekly; tighten budget to 25k if overshoot > 10% sustained.

### Test strategy

- **Unit:** Vitest on new modules (`generate.ts`, `runpod-text.ts`, `sse.ts`, extended `rate-limit.ts`)
- **Integration:** Mock upstream `fetch` to simulate RunPod SSE responses (including edge cases: missing `[DONE]`, connection-drop mid-stream, JSON-corrupted frame)
- **Contract:** Verify response shapes match `gemma-gateway-decision.md` §4 schemas
- **Regression:** Existing test suite unchanged; rerun to prove zero side effects
- **E2E smoke:** Live curl against production endpoint after deploy (runbook in `docs/usage/text-gen-deploy.md`)

### Handoff to Story 4.3

On successful close of Story 4.2:
- Text endpoint is live, `/v1/generate` responds with SSE
- Story 4.3 can transition Draft → Ready
- Story 4.3 owns SDK v0.4.0 additions (`complete()` + `completeStream()`), examples (Colab, CLI, curl), error taxonomy, alpha-access opt-in field
- Naming decision deferred: SDK surface for text (we won't rename existing `generate()` image method — per ADR-0004 Implementation Notes, use `complete()` for text OR namespace as `client.text.complete()`. Final TBD in Story 4.3 by @dev + @ux-design-expert.)
