## Story

**As** a servegate consumer (developer integrating i2i into a product in the portfolio),
**I want** a new `client.edit({ prompt, image, ... })` SDK method that sends `(input image + text instruction)` through the existing async submit/poll contract and receives an edited image inline in the response,
**so that** I can build image-editing features in my apps under a clean Apache 2.0 license with self-hosted inference economics (~$0.009/image at scale) — without per-image vendor invoices, without vendor TOS revocation risk, and without requiring any new infrastructure beyond what servegate already runs for text-to-image.

## Acceptance Criteria

1. **Handler dispatch by payload shape** — `serverless/handler.py:handler()` routes requests based on presence of `input_image_b64` field in the input payload: present → new Qwen-Image-Edit workflow branch; absent → existing FLUX.1-schnell T2I workflow (unchanged byte-for-byte). Branching logic is deterministic, covered by unit tests, and documented inline with ADR-0003 reference.

2. **Qwen workflow graph implemented** — new function `build_qwen_edit_workflow(prompt, input_image_b64, strength, seed, steps)` returns a ComfyUI graph that integrates: `QwenImageEditDiffusionModelLoader` (UNet fp8) + `QwenImageEditCLIPLoader` (Qwen2.5-VL encoder) + `QwenImageEditVAELoader` + `LoadImageBase64` (custom node decoding input_image_b64) + `VAEEncode` (input → latent) + `LoraLoader` (Lightning 8-step LoRA) + `CLIPTextEncode` (positive + negative conditioning) + `KSampler` (denoise = strength, 8 steps default) + `VAEDecode` + `SaveImage`. Graph is validated against the reference workflow in `serverless/workflow_template_qwen_edit.json` (new file, additive to existing `workflow_template.json`).

3. **Input validation rules enforced** — `normalize_input` in the i2i branch:
   - Rejects exact 1:1 aspect ratio (H == W) with HTTP 400 and error `invalid_aspect_ratio`, per documented gotcha (GitHub issue #243 in Qwen repo — 1:1 degrades coherence)
   - Downsamples inputs > 1 megapixel (H × W > 1,048,576) to ≤ 1 MP preserving aspect ratio (handler-side defensive; SDK also catches client-side)
   - Rejects payloads > 8 MB decoded (base64 inflates to ~11 MB on wire; 8 MB decoded cap keeps request body comfortably under Cloudflare Workers soft limits)
   - Rejects unsupported MIME types (only `image/png`, `image/jpeg`, `image/webp` accepted — verified via magic-byte check on decoded bytes, not Content-Type header trust)
   - `strength` parameter validated as float in `(0.0, 1.0]`, default 0.85
   - `steps` parameter default 8 (Lightning LoRA), accepts 4-50 range

4. **Output post-processing** — Qwen's known aspect-ratio drift (HF discussion #11) is mitigated by resizing the generated image to match the input's dimensions (width × height) before base64-encoding. Resize uses Pillow `LANCZOS` for quality. Output metadata includes original Qwen-generated dimensions alongside the resized-to-input dimensions for transparency.

5. **SDK v0.3.0 `edit()` method — strictly additive** — `sdk/src/client.ts` adds `edit(input: EditInput): Promise<GenerateOutput>` which:
   - Accepts `image: Buffer | Uint8Array | Blob | string` (base64 string)
   - Normalizes binary formats to base64 via the existing Node polyfill layer
   - Validates client-side: size ≤ 1 MP (downsample via `sharp` if dev opt-in; otherwise throws `ValidationError` with actionable message), rejects 1:1 aspect ratio with `ValidationError`
   - POSTs to `/jobs` with `{ prompt, input_image_b64, strength?, seed?, steps?, aspect_ratio? }` — payload shape triggers handler's i2i branch
   - Returns the same `GenerateOutput` shape as `generate()` (image_b64 inline per AD-1)
   - Reuses the existing submit/poll/retry machinery; no new polling logic
   - Exports new `EditInput` type from `sdk/src/types.ts`; existing `GenerateInput` + `GenerateOutput` + all typed errors unchanged
   - Existing `client.generate()` callers see zero behavioral or type-signature change (strict additivity verified via TypeScript `tsc --noEmit` + vitest regression suite)

6. **T2I regression = zero** — Existing `POST /jobs { prompt, steps, seed, width, height }` (no `input_image_b64`) produces:
   - Byte-identical `image_b64` output for a fixed seed (verified via SHA-256 hash comparison in a regression test)
   - Same latency envelope (~3.5s warm ± 10% at 1024² with 4 steps)
   - Same RunPod cost per image (no change to schnell workflow)
   - Same error behavior (all T2I error codes unchanged)
   - Async contract identical (ADR-0002 unchanged — POST /jobs returns 202 + job_id + status_url; GET /jobs/{id} returns 200 + output | 202 queued/running | 404 | 500 | 504)

### Strategic Decisions (from ADR-0003)

1. **Self-hosted Qwen-Image-Edit (Apache 2.0) over hosted APIs** — portfolio reuse + per-image economics drove rejection of BFL FLUX.2 [pro] ($0.045/img) and Gemini 2.5 Flash Image ($0.0395/img). See ADR-0003 §3.
2. **Lightning 8-step LoRA as default acceleration** — drops inference from ~71s to ~34s warm (community-reported on RTX 4090D). Non-accelerated 50-step path available as `steps: 50` opt-in for quality-sensitive requests (~2× cost: ~$0.019/img vs ~$0.009/img).
3. **Multi-provider gateway shape preserved** — this story does NOT build the BFL fallback; but code changes do not remove the ability to add it as secondary upstream later. `recommended-approach.md` §4.2 confirms gateway stays opaque-passthrough.
4. **Branch-by-payload dispatch** — handler.py dispatches based on presence of `input_image_b64`; gateway remains fully opaque (CON-3). Cleaner than a new endpoint path.
5. **Output inline base64 (AD-1 preserved)** — no signed-URL handling needed because inference runs on our compute; base64-encode and return inline like T2I does today.

### Implementation Sketches (from ADR-0003 §"Implementation Notes")

**handler.py i2i dispatch:**
```python
def handler(job):
    params = normalize_input(job.get("input") or {})
    if params.get("input_image_b64"):
        workflow = build_qwen_edit_workflow(**i2i_params(params))
    else:
        workflow = build_t2i_workflow(**t2i_params(params))  # unchanged
    # rest (queue_workflow, poll_history, extract_image_bytes) unchanged
```
