## Story

**As a** owner do Epic 2 precisando definir SLA do endpoint para consumers downstream (Story 2.2 SDK, Story 2.3 demo web),
**I want** decisão formal sobre como mitigar o cold-start de ~150s medido na Story 2.1 (Opção B bake-in vs workersMin=1 standby vs accept-as-is com disclaimer),
**so that** a documentação do Epic 2 e o contrato do SDK TypeScript reflitam SLA realista, e consumidores finais tenham expectativa calibrada.

## Acceptance Criteria

1. **Análise comparativa documentada** em `docs/architecture/adr-0001-flux-cold-start.md` com pelo menos 3 paths avaliados:
   - **A** (status quo): aceitar cold ~150s, documentar claramente no SDK/UI, adicionar warmup call opcional pra clients
   - **B** (bake-in models): imagem ~35GB, cold ~15-25s, +~10min de build time, sem network volume dependency
   - **C** (workersMin=1 standby): cold quase zero pro primeiro worker, custo adicional ~$0.66/dia ≈ $20/mês 24/7 idle
   - (opcional **D**) híbrido: workersMin=1 durante horário comercial apenas (scheduled scaling se RunPod suportar)

2. **Decisão registrada** no ADR com:
   - Path escolhido (1 das 3-4 opções)
   - Critérios de decisão usados (custo mensal, UX impacto, complexidade operacional, reversibilidade)
   - Data da decisão, responsáveis (@architect propõe, @pm aprova)
   - Plano de reversão se evidence contradizer depois de 30 dias de produção

3. **SLA atualizado** em `docs/prd/epic-2-consumer-integration.md` com cold-start real do path escolhido e expectativa comunicada aos consumers (por exemplo: "cold first-invocation após 5+ min ocioso = ~15-25s; warm subsequentes = <10s p95").

4. **Impacto em Stories downstream avaliado:**
   - Story 2.2 (TS SDK): decide se implementa warmup helper, retry com backoff em cold, ou expõe `isWarm` check
   - Story 2.3 (demo web): decide se UI mostra "warming up..." spinner ou assume sempre warm
   - Story 2.5 (gateway Cloudflare): decide se gateway pré-aquece worker antes de forward

5. **Story filha criada** (se decisão requer implementação — paths B, C, D):
   - Se A (accept): nenhuma story filha; fecha 2.1.2 como "decision only"
   - Se B (bake): `docs/stories/2.1.3.bake-models-into-image.story.md` drafted por @sm
   - Se C (standby): `docs/stories/2.1.3.enable-workers-min-standby.story.md` drafted por @sm
   - Se D (hybrid): story + dependency em RunPod scheduled scaling feature

**Task 1 — Data gathering findings:**

*Cold bench analysis:*
- n=2 true cold spawns (out of 5 attempts): 115s wall + 150s wall (median 133s)
- 2 hits warm (flashboot retention >180s idle) — não contam como cold
- 1 failure executionTimeout 171s (pré-patch 120→300s)

*Pricing confirmed:*
| GPU | Flex | Active | 24/7 active cost |
|---|---|---|---|
| RTX 4090 24GB | $0.00031/s | $0.00021/s | **$544.43/mo** |
| RTX A5000 24GB | $0.00019/s | $0.00013/s | **$337.02/mo** |

*Critical hypothesis validated:* `workersMin=1` é billed CONTINUAMENTE no rate "active" (não flex). Path C tem custo fixo floor independente de volume.

*Other validations:*
- FLUX.1-schnell é license-gated (requires HF_TOKEN em CI/build) — adiciona complexidade para Path B
- RunPod sem scheduled scaling nativo — Path D requer external cron (3-5 dev-days)

**Task 2 — ADR decisão:**

**Verdict: Path A (Accept cold, solve via consumer UX)** — recomendado ao @pm para aprovação.

Rationale comprimido:
- Para volume MVP (100-500 imgs/mo), Path A custa $3-8/mo vs Path B $0.60-2/mo — delta $5/mo não justifica 1-2 dev-days + AC2 violation (35GB image)
- Path C economicamente inviável — $337-544/mo floor independente de volume (reject)
- Path D complexidade alta para economia marginal (reject para MVP)
- Revisão 30 dias pós-approval + critério reversibilidade documentado
