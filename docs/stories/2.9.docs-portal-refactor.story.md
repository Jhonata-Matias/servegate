## Story

**As a** developer evaluating servegate (or alpha consumer integrating the SDK),
**I want** a navigable, search-enabled docs portal at `deploy-lp-one.vercel.app/` instead of a single-page marketing landing,
**so that** I can quickly find the API reference for the endpoint I'm calling, jump between SDK methods and their error classes, and discover new features (like multi-image i2i shipped in PR #15) without scrolling 995 lines of mixed marketing+reference content.

**Trigger:** PR #15 merged (multi-image i2i via Qwen-Image-Edit 2509). Current landing's API/SDK references are outdated AND structurally undersized — adding multi-image inline would push the file past 1100 lines while keeping the single-page constraint that already harms discoverability.

**Pattern reference:** RunPod docs portal (https://docs.runpod.io) — sidebar nav · centered content · TOC right · search · card-grid Welcome.

## Acceptance Criteria

1. **Astro Starlight project initialized** at `web/docs/` with TypeScript strict mode, MDX content support, and Pagefind search built-in. Astro auto-detected by Vercel (no `vercel.json` needed unless overriding build root).

2. **5 docs pages migrated** from `web/landing/index.html` content + new content for multi-image i2i:
   - `src/content/docs/index.mdx` — **Welcome** with custom card-grid 3×2 (Quickstart · Authentication placeholder · API Reference · SDK · Errors · Releases) replicating RunPod ref pattern
   - `src/content/docs/quickstart.mdx` — **Quickstart** with 3 tabbed examples (T2I · i2i 1-image · i2i 2-image NEW)
   - `src/content/docs/api.mdx` — **API Reference** with all current endpoints + new `input_image_b64_2` field documented in i2i request body table
   - `src/content/docs/sdk.mdx` — **SDK (TypeScript)** referencing `@jhonata-matias/flux-client@0.4.0` with `edit()` single-image AND multi-image examples + `field: 'image2'` discriminator on validation errors
   - `src/content/docs/errors.mdx` — **Errors** consolidating HTTP errors (401/429/502/504) + i2i_validation variants (incl. multi-image error tagging)

3. **Sidebar navigation** matches the IA in the spec: 3-level hierarchy (group → page), active state highlights current page, configured via `astro.config.mjs` Starlight `sidebar` array. No empty placeholder pages — defer Authentication / Concepts / Releases / Resources to V2 backlog.

4. **Brand identity preserved.** Theme overrides at `src/styles/theme.css` map current landing palette to Starlight CSS custom properties: charcoal background (`#0a0a0a`), zinc-9xx surfaces, teal accent (exact hex must be sourced from `web/landing/index.html` Tailwind config to avoid drift), Geist Sans + Geist Mono. Dark-only at V1 (light theme deferred to V2 backlog).

5. **Multi-image i2i is documented in 3 places** (API · SDK · Errors). Each mention is concrete:
   - API: new optional row `input_image_b64_2` in i2i request body table.
   - SDK: working `edit({prompt, image, image2})` example + version bump callout.
   - Errors: explicit mention of `field: 'image2'` discriminator in `i2i_validation` error responses.

6. **Pagefind search** indexes all 5 pages and returns relevant results for queries: `multi-image`, `image2`, `polling`, `rate limit`. Verified manually post-build via `pnpm build && pnpm preview` then Pagefind UI in browser.

7. **Lighthouse score ≥ 90** on each of the 5 pages (Performance · Accessibility · Best Practices · SEO). Captured via Chrome DevTools or `lighthouse` CLI against the Vercel preview URL. Reports archived under `docs/qa/lighthouse-2.9/`.

8. **WCAG AA contrast** verified on all text against the charcoal background. Tools: axe DevTools or `@axe-core/cli` against the preview URL. Zero contrast violations.

9. **No broken internal links** post-build. Tool: `pnpm dlx linkinator dist/ --recurse --silent --skip 'github.com'` (skip external GitHub since rate-limited). Output saved to `docs/qa/2.9-linkinator.txt`.

10. **Vercel cutover validated.** Preview deploy URL shared for `@ux-design-expert` + user review BEFORE production cutover. `@devops` switches Vercel project root from `web/landing/` → `web/docs/` only after explicit approval. Old landing moved to `web/_archive/landing-pre-docs-2026-04-29/` (preserves git history; user can purge after 30 days of stable docs portal).

### Special Execution Note: Vercel cutover (Phase 4)

This is the **only critical-path remote operation** in the story. Order of operations:

1. `@dev` completes Phases 1-3, commits, runs CodeRabbit clean.
2. `@dev` notifies `@devops` story is ready for PR.
3. `@devops` runs `*pre-push` quality gates, opens draft PR.
4. Vercel auto-deploys preview from PR branch (uses current `web/landing/` root since project setting hasn't changed yet — preview URL will show OLD landing). **This is expected — Phase 4 cutover happens AFTER preview validation of the docs build.**
5. `@dev` builds docs locally (`cd web/docs && pnpm build`) → upload `dist/` to a temporary Vercel project OR use Vercel CLI with `--cwd web/docs` for direct preview deploy of the new docs.
6. Share temporary preview URL with user + `@ux-design-expert` for review.
7. On approval: `@devops` updates Vercel project root setting (`web/landing/` → `web/docs/`) AND merges the PR. Both happen in tight sequence to minimize the window where main has new code but Vercel still serves old landing.
8. Verify production: visit `deploy-lp-one.vercel.app/` and confirm new docs portal renders.
9. If broken: `@devops` reverts Vercel root setting (single-click rollback in Vercel UI) — main commit doesn't need revert because old landing files still exist at `web/_archive/`.
