## Story

**As a** owner do projeto gemma4 protegendo budget e prevenindo abuse,
**I want** um Cloudflare Worker que serve como **único** ponto de entrada público pra geração de imagens — autenticando via `X-API-Key`, aplicando rate-limit global de 100 imagens/dia, e proxiando requests autorizados pro RunPod Serverless endpoint (Story 2.1) — com response 429+Retry-After quando o limite é atingido,
**so that** consumidores TypeScript (Story 2.2) e a app web demo (Story 2.3) tenham uma URL HTTPS estável, autenticada, com cap de custo determinístico (~$1.80/mês máx), e o RUNPOD_API_KEY nunca seja exposto a clientes.

## Acceptance Criteria

1. **Worker deployado:** Cloudflare Worker em `workers.dev` (free) ou custom domain, recebendo `POST /` (ou `POST /generate`) com header `X-API-Key` e body JSON do mesmo schema do endpoint serverless (Story 2.1).
2. **Auth middleware:** valida header `X-API-Key` contra `GATEWAY_API_KEY` (Cloudflare Secret). Ausente ou diferente → HTTP `401 Unauthorized` com body `{"error": "invalid_api_key"}`. Retorna ANTES de incrementar rate-limit counter.
3. **Rate-limit middleware:** lê chave KV `count:YYYY-MM-DD` (UTC), incrementa atomicamente, retorna HTTP `429 Too Many Requests` com header `Retry-After: <segundos até 00:00 UTC>` e body `{"error": "rate_limit_exceeded", "limit": 100, "reset_at": "ISO-8601"}` quando contador `>= 100` ANTES do incremento (request 101 falha; request 100 ainda passa).
4. **KV TTL:** chave `count:YYYY-MM-DD` tem TTL de 48 horas (auto-cleanup). Novo dia (00:00 UTC) começa com chave inexistente = contador 0.
5. **Proxy autenticado:** request válido (auth ✅ + rate-limit ✅) é proxiado pra RunPod Serverless endpoint via `fetch` com header `Authorization: Bearer ${RUNPOD_API_KEY}` (Cloudflare Secret separado). RUNPOD_API_KEY **NUNCA** retorna ao cliente.
6. **Response passthrough:** resposta da RunPod Serverless (status, body JSON) é devolvida ao cliente sem modificação, exceto headers adicionais: `X-RateLimit-Remaining: <100-counter>`, `X-RateLimit-Reset: <ISO-8601>`.
7. **Smoke test 101 requests:** script faz 101 POSTs sequenciais com X-API-Key válido — primeiras 100 retornam status 200 (proxy serverless), request 101 retorna 429 com `Retry-After` header presente e valor correto (segundos até próxima meia-noite UTC).
8. **Logs estruturados (sem PII):** Worker loga 1 linha JSON por request com `{timestamp, ip, status, elapsed_ms, day_count}`. **NUNCA** loga prompt content nem image bytes (LGPD/privacy). Visível em `wrangler tail` e Cloudflare dashboard.

### KV Eventual Consistency — Trade-off documentado

CF Workers KV tem **eventual consistency global** (até ~60s pra propagar entre regiões). Em prática:
- Single user/region: leitura imediata após write (read-your-writes)
- 2 requests simultâneos no mesmo ms: ambos podem ler `count=99`, ambos incrementam pra `100` = **request 101 ainda passa** (overshoot 1)
- Worst case observado: 1-2 imgs/dia além do limite

**Aceito per Epic 2 R7.** Atomicidade rigorosa requer Durable Objects (mais complexo, custa mais — escalar pra @architect se necessário).

### Code Quality Assessment

**Pontos fortes:**

1. **Clean 4-phase orchestration** em `index.ts:fetch()`: método → auth → rate-limit → proxy — cada fase falha fast com early return + logging específico
2. **Auth executa ANTES de KV read** — economia de KV quota para requests inválidas; também evita auth timing leak via KV latency
3. **`constantTimeEqual`** XOR-loop com length-safe dummy loop (evita length-leak timing attack via branching)
4. **Module separation**: 5 files focused (types, auth, rate-limit, proxy, log, index) — cada um testável isoladamente
5. **JSDoc inline** em cada módulo com contract + trade-offs documentados (R7 explícito)
6. **Conditional spread** em error responses (e.g., `...(body?.error ? { message: body.error } : {})`) respeita `exactOptionalPropertyTypes`
7. **AbortController + clearTimeout** em proxy.ts (60s RunPod timeout) — zero timer leak
8. **Error mapping** upstream: 5xx → 502 (não passthrough para não expor internals), 4xx → passthrough (validação errors are consumer concerns)
9. **Rate-limit state returned to caller** — headers adicionados em success path via `proxy.ts` sem acoplar à rate-limit.ts
10. **Private fields / immutable config** em rate-limit constants (`DAILY_LIMIT`, `KV_TTL_SECONDS`)
