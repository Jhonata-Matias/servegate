## Story

**As a** owner do projeto gemma4,
**I want** empacotar o workflow ComfyUI + FLUX.1-schnell como um endpoint RunPod Serverless funcional, com handler Python que aceita JSON de input e retorna a imagem gerada,
**so that** consumidores TypeScript (Story 2.2) e a app web demo (Story 2.3) possam chamar uma URL HTTPS estável que escala automaticamente, sem o custo de Pod 24/7 self-hosted, com cold start <30s e warm latency <10s.

## Acceptance Criteria

1. **Handler Python:** `handler.py` implementa interface `runpod.serverless.start({"handler": fn})`, aceita input JSON `{"prompt": str, "steps": int (default 4), "seed": int (default random), "width": int (default 1024), "height": int (default 1024)}`, retorna `{"image_b64": str, "metadata": {"seed": int, "elapsed_ms": int}}`
2. **Dockerfile:** builda imagem CUDA-ready com ComfyUI + Python deps + handler. Imagem otimizada (multi-stage se possível) e <15 GB final.
3. **Modelos:** estratégia escolhida e justificada (bake-in image vs network volume vs download-on-cold-start) — qualquer escolha que mantenha cold start <60s.
4. **Deploy:** endpoint RunPod Serverless criado via dashboard ou API, com template ID e endpoint ID documentados em `.env` (gitignored). Test request real retorna imagem PNG válida.
5. **Latência medida:** warm <10s p95, cold <30s p95 — em **100 invocações reais sequenciais** (warm) e **5 cold spawns** (forçados por idle wait). Métricas registradas no Dev Agent Record.
6. **Custo medido:** custo real de 100 invocações documentado e comparado ao projetado ($0.06 esperado per cost projection do Epic 2). Documentação em `docs/usage/runpod-serverless-deploy.md`.

4. **`/runsync` vs `/run`:** Smoke test inicial usando `/runsync` retornou `IN_QUEUE` em cold start (HTTP client timeout antes de cold init completar). Pivot: smoke.sh e latency_bench.py migrados para `/run` + polling `/status`.

**AC5 Gap — Cold Latency (DOCUMENTED MISS):**
- **Target:** cold p95 <30s
- **Medido:** cold p95 ~115-150s (n=2 true colds de 5 tentativas; 3 hit workers warm via flashboot)
- **Root cause:** Opção A escolhida (network volume) requer load do UNet FLUX schnell 23GB FP16 via NVMe network (200-400 MB/s throughput). Load time isolado: 60-90s. Adicional: ComfyUI boot + CLIP + VAE load + first inference warmup = 30-60s. Total cold = ~1-3min cold start wall.
- **Impacto:** warm path (contrato primário para Story 2.2 SDK e Story 2.3 demo) **passa AC5 com folga** (p95=7.0s vs target <10s). Cold path afeta primeira invocação após longo idle.
- **Mitigações possíveis (recomendações futuras):**
  - Pivotar para Opção B (bake-in models) — imagem ~35GB mas cold ~15-25s
  - Ou `workersMin: 1` standby (custo ~$16/mês 24/7) para eliminar cold starts
  - Ou flashboot mais agressivo (já `true`; parece manter workers ~180s+ idle sem descarga)
- **Decisão:** NÃO pivotar nesta story. Warm meets AC5 primary use case (app/SDK tráfego contínuo). Cold gap registrado para decisão Epic 2 downstream.

**AC6 Gap — Cost vs Projection (DOCUMENTED DISCREPANCY):**
- **Target:** $0.06/100 invocações (projeção Epic 2)
- **Medido:** ~$0.0003/s measured confirmado (RTX 4090 flex rate); warm @ ~5s/image ⇒ ~$0.0015/image ⇒ ~$0.15/100 invocações
- **Discrepância:** 2.5x over projection
- **Root cause:** Epic 2 assumiu ~1s/image (aplicável a SDXL Turbo ou 512×512 low-step). Realidade FLUX schnell 1024×1024 4-step em RTX 4090 = 3-5s inferência warm.
- **Impacto:** ordem de grandeza correta ($0.01 vs $0.10 por imagem, não $0.01 vs $1.00). Economia massiva vs Pod 24/7 ($500/mês se 1 pod ligado 24h × 30d) permanece — Serverless é ~$50/mês para uso esporádico estimado em $100-500/mês de imagens.
- **Decisão:** atualizar projeções em Epic 2 PRD futuro. Sem ação nesta story.

### Issues Found

#### 🟡 MEDIUM (recomenda fix antes de Done)

**M1 — Cold latency 5x acima do target AC5**
- **Severidade:** Medium (affects Epic 2 consumer UX para primeira request)
- **Evidência:** Cold spawns reais medidos em 115s e 150s (n=2); target <30s; também afeta AC3 <60s implícito
- **Root cause:** Opção A (network volume) requer load do UNet FLUX.1-schnell 23GB via NVMe network (200-400 MB/s) = 60-90s só para modelo
- **Status:** Bem documentado em Dev Agent Record com 3 mitigation paths propostos
- **Recomendação:** NÃO bloquear merge. Criar backlog item Epic 2 para decidir entre (a) pivot Opção B bake-in, (b) workersMin=1 standby, (c) accept as-is. Decisão deve considerar orçamento ($16/mo standby vs ocasional cold UX)

**M2 — Config inconsistente: deploy.sh executionTimeoutMs=120s, live endpoint=300s**
- **Severidade:** Medium (reprodutibilidade quebrada)
- **Evidência:** `serverless/deploy.sh:90` hardcode `executionTimeoutMs: 120000`; endpoint live patched para 300000 durante T8 após cold #2 timeout; se alguém reaplica deploy.sh idempotente, reverterá a configuração problemática
- **Recomendação:** Atualizar deploy.sh para `executionTimeoutMs: 300000` (documentado também em docs/usage para coerência)

#### 🟢 LOW (fix opcional, podem virar tech debt)

**L1 — `normalize_input()` silenciosamente corrige valores 0 para defaults**
- **Evidência:** `steps = int(raw.get("steps") or DEFAULT_STEPS)` — `steps=0` passa o validador porque `0 or 4 == 4`; também afeta width/height
- **Impacto:** Violação silenciosa de user intent; baixa porque defaults são sensatos (1024×1024, 4 steps)
- **Fix sugerido:** 
```python
v = raw.get("steps")
steps = DEFAULT_STEPS if v is None else int(v)
```

**L2 — `wait_for_comfy` captura `Exception` broadly**
- **Evidência:** handler.py:70 `except Exception as e:` durante polling de readiness
- **Impacto:** Pode mascarar bugs reais no start.sh (ex: PermissionError, SystemExit)
- **Fix sugerido:** Limitar a `(urllib.error.URLError, ConnectionError, TimeoutError)`

**L3 — `COMFY_GENERATION_TIMEOUT_S=120` default é curto para cold primeira inferência**
- **Evidência:** handler.py:30 default 120s; cold primeira inferência pode exceder (observado server=98s mais HTTP retry)
- **Fix sugerido:** aumentar para 180s ou alinhar com `executionTimeoutMs` via env

**L4 — `poll_history` não termina em status_str não previstos**
- **Evidência:** handler.py:130-141 só para em "completed" OU "error"
- **Impacto:** Stall até COMFY_GENERATION_TIMEOUT_S se ComfyUI retornar status novo/inesperado
- **Fix sugerido:** log warning para status_str desconhecidos, seguir polling

**L5 — Ausência de unit tests**
- **Status:** aceitável conforme story ("integration-heavy") mas reduz confidence em refactors futuros
- **Recomendação:** Adicionar `tests/test_normalize_input.py` cobrindo os 10+ edge cases que QA já exercitou
