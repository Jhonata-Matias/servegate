## User Story

**As** the servegate maintainer (and any future portfolio app that needs text generation),
**I want** a formal Phase 0 foundation that fixes which Gemma variant, runtime, GPU tier, gateway integration, and rate-limit strategy the text-gen capability will use,
**so that** Story 4.2 (gateway + worker implementation) and beyond have a non-invented, cost-validated, license-audited decision to execute against — and portfolio-wide cost never gets surprised by a silent over-spec.

---

## Context

This story formalizes the research and architectural decisions needed before writing any text-gen production code. The work breaks into four Phase 0 deliverables (per `wf-gated-model-serve` step 0.1–0.4) plus a cost model; most content has already been drafted alongside the epic in this branch. Story 4.1's real job is to **QA-gate the artifacts** against the workflow's `FOUNDATION_SET` checkpoint (5 specific criteria) and drive ADR-0004 from Proposed → Accepted.

No production code changes in this story. No gateway changes. No serverless endpoint changes. No SDK changes. Image-gen stack (Epics 1–3) stays byte-identical.

## Dev Notes

### Starting point
Most of the content for AC1–AC5 was drafted during epic preparation (see References at the bottom). Story 4.1 is largely a formalization + QA pass, not from-scratch research. If any section fails completeness check, extend in place — don't rewrite.

### ADR-0004 Status transition rule
- Proposed: written, awaiting validation
- Accepted: Story 4.1 QA gate PASS with GO verdict on all 5 GMS_FND_001 criteria
- Superseded: only via a future ADR that explicitly replaces this one

### Source validation for research docs
All external sources cited must:
- Resolve (no 404)
- Be dated ≤ April 2026 (our decision horizon)
- Be reachable without authentication (public docs / blogs / papers only — no paywalled sources)

### Phase 0 gate constitutional trace
Per `wf-gated-model-serve` Article IV (No Invention): every architectural claim in ADR-0004 MUST trace to one of:
- A functional requirement (FR-) in this epic
- A non-functional requirement (NFR-) implied by the <alpha cost ceiling> or 180s cold-start budget
- A constraint (CON-) documented in gateway-decision or provider-assessment
- A research finding from model-candidates or provider-assessment or external source citation

If any ADR statement lacks a trace, flag in QA gate as `NEEDS_REVISION`.

### Handoff to Story 4.2
On successful close of Story 4.1:
- ADR-0004 becomes Accepted
- Story 4.2 can transition Draft → Ready (unblocked)
- Story 4.2's first task is the SSE pass-through spike (R1 from gateway-decision) — validate `new Response(upstream.body, ...)` against a real RunPod vLLM endpoint before any further implementation work.
