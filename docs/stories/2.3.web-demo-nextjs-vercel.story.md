## Story

**As a** stakeholder ou primeiro usuário visitando o demo do gemma4 para avaliar capability de geração FLUX,
**I want** uma página web Next.js deployada em Vercel que pré-aquece o endpoint em background no page-load, mostra progresso explícito durante o warmup (~130s cold first-time), aceita um prompt + params, e retorna a imagem gerada consumindo o SDK `@jhonata-matias/flux-client` via gateway autenticado,
**so that** a primeira impressão do projeto não é um botão "Generate" travado por 2 minutos sem feedback, eu possa avaliar qualidade de geração em <3min total, e meus eventos de interação sejam medidos (analytics) para o 30-day review gate (2026-05-21) decidir pivot Path B.

## Acceptance Criteria

> Derivados de `docs/prd/epic-2-consumer-integration.md` v0.4 — seção "Preliminary ACs — Story 2.3 — Next.js Web Demo". Ancorados em ADR-0001 PM Addendum 4 (4 UX mitigations) + SLA v0.3 4-layer table.

1. **Pre-warm on page visit** — `useEffect()` on page mount dispara `client.warmup()` em **background** imediatamente (< 500ms após page load), SEM esperar click do botão "Generate". Botão permanece disabled (ou enabled com visual cue) até warmup resolver. Zero cliques necessários para iniciar o warmup.

2. **Explicit progress messaging** — UI mostra texto explícito durante cada estado (NÃO spinner genérico):
   - `warming` (warmup em curso): `"Preparing server (one-time setup, up to 2 min)..."` + **timer visível** contando segundos decorridos
   - `ready` (warmup done): `"Ready to generate"` + botão enabled
   - `generating` (request em curso): `"Generating image..."` + timer
   - `done` (imagem retornada): imagem renderizada + `"Generate another"` action
   - `error`: mensagem específica por tipo de erro (ColdStartError → "Server taking longer than expected, try again"; RateLimitError → "Rate limit, try in {X}s"; AuthError → "Configuration issue, contact admin")

3. **Demo state save** — SessionStorage persiste `{ lastPrompt, lastImageB64, lastGeneratedAt }` entre reloads. Se usuário recarrega a página (F5), estado último visível restaura SEM re-gen automático. Se usuário sai e volta < 5min (warm window), `client.isWarm()` retorna true, pulando warmup → UX instant-ready.

4. **Day-1 analytics** — Emite 4 eventos observáveis (via Vercel Analytics custom events OU POST para logging endpoint do gateway):
   - `warmup_start` → `{ timestamp }`
   - `warmup_end` → `{ duration_ms, was_cold }`
   - `first_interaction_latency` → `{ page_load_ms_to_first_image_rendered }`
   - `bounce` → `{ reason: 'abandoned_warmup' | 'abandoned_before_first_image' | 'session_timeout' }` (emit em `window.beforeunload` se user sai sem concluir fluxo)
   - Analytics dashboard (Vercel Analytics OU custom view) deve mostrar ≥1 evento de cada tipo após 5 test sessions.

5. **Gateway auth flow (server-side)** — `X-API-Key` e `GATEWAY_URL` embedded **exclusivamente** em Next.js server-side (route handler `/api/generate` via App Router). **ZERO** ocorrências de API key no client JS bundle. Validação empírica: `grep -r "GATEWAY_API_KEY\|<actual-key-value>" .next/static/` retorna 0 matches após build. 429 response propagado ao client com `retry_after_seconds` (exposto pela SDK `RateLimitError`).

6. **Vercel runtime config** — Route handler `/api/generate` configurado com `export const runtime = 'nodejs'` (não `'edge'`) para permitir timeouts 60s+ (cold start até 180s). Comentário inline explicando razão (ADR-0001 Path A cold ceiling). `maxDuration` configurado para 300s (ou limite do Vercel Pro se necessário no futuro; free tier suporta até 60s — **validar em Task 1 se free tier basta para cold path**).

7. **First-load-to-interactive SLA** — Meta: **p95 < 150s** (page_load → first image rendered) em 10 sessões controladas de smoke test. Medido via evento `first_interaction_latency` (AC4). Meta stretch para 30-day review (2026-05-21): **p95 < 120s** (indica pre-warm working well). Acceptance: 9/10 sessões abaixo de 150s total.

8. **Accessibility + responsive** — Lighthouse (Chrome DevTools, mobile viewport) Accessibility score ≥ 90. Requisitos mínimos:
   - Semantic HTML (`<main>`, `<h1>`, `<form>`, `<button>`)
   - `aria-live="polite"` no status messaging (AC2) para screen readers
   - Keyboard navigation funcional (Tab, Enter, Space)
   - Contrast ratio WCAG AA (>= 4.5:1 para texto normal)
   - Responsive 320px viewport (mobile) até 1920px (desktop)
   - Visible focus indicators

### UI State Machine

```
        ┌─────────┐
        │  idle   │ (SSR initial, pre-hydration)
        └────┬────┘
             │ useEffect mount
             ▼
        ┌─────────┐
   ┌───►│ warming │ (timer visível, "Preparing server...")
   │    └────┬────┘
   │         │ warmup success
   │         ▼
   │    ┌─────────┐
   │    │  ready  │ (button enabled, "Ready to generate")
   │    └────┬────┘
   │         │ user submit
   │         ▼
   │    ┌───────────┐
   │    │generating │ (timer, "Generating image...")
   │    └────┬──────┘
   │         │
   │         ├─ success ──► ┌──────┐
   │         │              │ done │ (image + "Generate another")
   │         │              └──┬───┘
   │         │                 │ reset click
   │         │                 └─────────┐
   │         │ error                     │
   │         ▼                           │
   │    ┌───────┐                        │
   └────┤ error │ (specific message)     │
        └───┬───┘                        │
            │ retry click                │
            └────────────────────────────┘
```
