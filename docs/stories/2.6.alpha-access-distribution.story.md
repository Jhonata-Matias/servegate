## Story

**As a** dev externo que descobre o repositório gemma4 e quer gerar imagens FLUX via API,
**I want** encontrar um README raiz com ponto de entrada claro, templates de issue formalizados para pedir acesso, contact channel público com SLA, SDK npm público instalável sem grant manual, API reference consolidada, e uma release v0.1.0-alpha taggueada como linha de base,
**so that** a jornada "descobri o repo → fiz primeira chamada funcionando" aconteça em 15-20 minutos sem hand-holding manual do owner, destravando distribuição alpha para os primeiros 3-5 devs convidados antes do beta.

## Acceptance Criteria

1. **SDK public visibility** — `@jhonata-matias/flux-client` muda de `visibility: private` → `public` em GitHub Packages. Em VM/container limpa, `npm view @jhonata-matias/flux-client --registry=https://npm.pkg.github.com` retorna metadata **sem auth**; `.npmrc` + `npm install` funciona com qualquer GITHUB_TOKEN tendo scope `read:packages`. README e dev-onboarding atualizados removendo (se aplicável) step manual "owner adiciona collaborator".

2. **Root README.md** — `README.md` na raiz do repo com estrutura:
   - Título + tagline ("gemma4 FLUX API — alpha image generation via authenticated gateway")
   - Badges: status=alpha, license=MIT, gateway=live
   - 3-bullet "What is this / For whom / Current status"
   - Quickstart: link direto para `docs/usage/dev-onboarding.md`
   - Seção Links: SDK, API reference, Terms, Privacy, Monitoring
   - Contact section (ver AC4)
   - License + footer
   Render no GitHub é a landing ao abrir `github.com/Jhonata-Matias/gemma4`.

3. **Issue templates** — `.github/ISSUE_TEMPLATE/` com 4 arquivos:
   - `access-request.yml` — GitHub Forms com campos required: `name`, `github_username`, `use_case` (textarea), `expected_volume` (dropdown: <10/dia, 10-50/dia, 50-100/dia), `tos_accepted` (checkbox required), `privacy_accepted` (checkbox required). Labels auto-aplicadas: `access-request`, `alpha`.
   - `bug-report.yml` — fields: `environment` (OS + Node version), `sdk_version`, `reproduction_steps`, `expected_behavior`, `actual_behavior`, `logs`.
   - `feature-request.yml` — fields: `problem_statement`, `proposed_solution`, `alternatives_considered`, `volume_impact`.
   - `config.yml` — `blank_issues_enabled: false` + `contact_links` apontando para dev-onboarding.

4. **Owner contact public** — seção "Contact" no `README.md` raiz + mirror em `sdk/README.md` documentando:
   - Canal preferido: GitHub issue via template `access-request.yml`
   - Fallback: GitHub DM ao owner (`@Jhonata-Matias`) ou email público (se owner quiser documentar)
   - SLA alpha: 3-7 business days first response
   - Expectativa: "personal project, best-effort support, no SLA guarantees during alpha"
   - Secure key delivery: método padrão documentado (GitHub DM + optional GPG key recebida via issue comment)

5. **GitHub Release v0.1.0-alpha** — tag git anotada `v0.1.0-alpha` apontando para commit `9d1f100` (merge Story 2.5), e GitHub Release (marked **Pre-release**) com notes:
   - Resumo Epic 1 (Pod Inference Stack Done) + Epic 2 (Consumer Integration in progress)
   - Deliverables shipped: RunPod Serverless endpoint, SDK v0.1.0, Cloudflare gateway, docs alpha
   - Links: SDK CHANGELOG, dev-onboarding, Terms, Privacy, API reference (AC6)
   - Known limitations: 100 imgs/dia global cap, ~130s cold start expected, invite-only access

6. **API reference consolidado** — `docs/api/reference.md` único documento cobrindo:
   - Gateway base URL + endpoint `POST /`
   - Request headers (`X-API-Key: required`, `Content-Type: application/json`)
   - Request body schema (JSON com `input.prompt`, `input.steps`, `input.width`, `input.height`, `input.seed`)
   - Response codes com payloads exemplo: 200 OK, 401 Unauthorized, 405 Method Not Allowed, 429 Rate Limited, 502 Bad Gateway, 504 Gateway Timeout
   - Rate-limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, `Retry-After`)
   - Curl quickstart end-to-end (copy-paste funciona)
   - Link cruzado para SDK README (TypeScript consumers) + Colab example (Python consumers)

7. **External smoke validation** — todos os 5 steps do `docs/usage/dev-onboarding.md` executados **em ambiente limpo por perfil diferente do owner** (segunda conta GitHub, colaborador convidado, ou container Docker zerado sem credenciais prévias): request key (via issue template) → install SDK (com `.npmrc` novo) → configure env → primeira chamada `generate()` → tratar error case (force 401 com key inválida OU deixar rate-limit hit). Documentar em `docs/qa/2.6-external-smoke.md` com timing por step + screenshots quando aplicável + gaps descobertos (não-bloqueantes viram follow-up issues, bloqueantes voltam para @dev fix).

### Strategic Decision — SDK public (PRD v0.7)

SDK `@jhonata-matias/flux-client` vai **public** porque:
- **Zero secret exposure** — gateway (Story 2.5) é quem guarda RUNPOD_API_KEY; SDK é client puro
- **Friction reduction** — alpha target 3-5 devs, grant manual escala ruim; public flip elimina step manual
- **Alpha invite-only preservado** — access control vive no GATEWAY_API_KEY issuance (manual review via issue template AC3), não no package visibility
- **Reversível** — flip `public → private` a qualquer momento via GH Packages UI

Beta futuro pode migrar para npm registry público oficial (complementar, não substituir GH Packages).
