## User Story

**As** a developer browsing servegate's public landing page (`gemma4-gateway.jhonata-matias.workers.dev` → linked from README and from `web/landing/`),
**I want** the landing page to document the new `POST /v1/generate` text-gen endpoint alongside the existing `POST /jobs` image-gen endpoint, with quickstart curls, error codes, and a corrected hero copy that mentions text generation as a first-class modality,
**so that** the public showcase reflects the full v0.3+ API surface, and so that two CodeRabbit-flagged inconsistencies on the same page are closed in the same pass instead of accumulating debt.

---

## Context

This is a polish/follow-up story to Story 4.2. The trigger is a gap discovered after Story 4.2 implementation completed:

- Story 4.2 shipped `POST /v1/generate` end-to-end (gateway, RunPod backend, AC11 production smokes PASS, QA Round 3 PASS 94/100), AND updated the **canonical** API reference at `docs/api/reference.md` plus a quickstart at `docs/usage/text-gen-api-quickstart.md`.
- Story 4.2 did NOT update `web/landing/index.html` because no AC required it. The landing page therefore documents only image-gen `/jobs` and is silent on text-gen, which leaves the public showcase incomplete from a user perspective.
- CodeRabbit pre-PR Round 3 also surfaced two pre-existing landing inconsistencies that were deferred at the time:
  1. I2I Validation list at `web/landing/index.html:531-537` is missing a documentation entry for the `invalid_image_base64` error code.
  2. Resources release-note copy at `web/landing/index.html:610-612` references "deploy serverless v0.2.0" while the heading shows "Release v0.3.0-alpha".

This story bundles these three items into a single landing-page polish pass to avoid three separate trivial PRs and to keep the public showcase coherent.

### PR situation at story creation

Story 4.2 PR #13 was **closed without merging** by the user on 2026-04-25 in order to reconverge the landing-page work onto the same branch (`feature/4.2-gemma-gateway-worker`) before any merge to `main` happens. Work on Story 4.2.1 should therefore be committed onto the same branch, and a fresh PR will be created at the end covering the combined Story 4.2 + 4.2.1 scope. See **Dev Notes — PR strategy** below.

## Acceptance Criteria

### AC1 — Text-gen API Reference block on landing

- [ ] `web/landing/index.html` API Reference section (`<section id="endpoints">`) contains a new `POST /v1/generate` block following the same visual pattern as the existing `POST /jobs` block.
- [ ] The block includes: HTTP method + URL, required headers (`Authorization: Bearer …`, `Content-Type: application/json`), a request body example using the chat-completion shape (`messages`, optional `model`, `stream`, `max_tokens`), a streaming response example (3–5 SSE frames including a final `data: [DONE]`), and a non-streaming response example matching the OpenAI `chat.completion` shape with `usage.total_tokens`.
- [ ] Error codes documented for `/v1/generate` include at least: `missing_messages`, `invalid_request`, `request_too_large`, `invalid_api_key`, `rate_limit_exceeded`, `upstream_error`, `upstream_unavailable`. Source for codes: `docs/api/reference.md` §`POST /v1/generate`.

### AC2 — Text-gen Quickstart on landing

- [ ] `web/landing/index.html` Quickstart section (`<section id="quickstart">`) gains a sub-section for text generation with two curl examples: streaming (`-N`, `stream: true`) and non-streaming (`stream: false`).
- [ ] Existing image-gen quickstart (T2I + I2I) preserved without any wording change.
- [ ] Code block styling matches existing siblings (same `code-block` class, same syntax-highlighting span pattern).

### AC3 — Errors section adds `invalid_image_base64` (CodeRabbit Round 3 deferred #4)

- [ ] I2I Validation list at `web/landing/index.html:531-537` adds an `<li>` entry for `invalid_image_base64` explaining "the `input_image_b64` field is not valid base64".
- [ ] Markup matches sibling entries (same classes, same tone).

### AC4 — Release-note version copy reconciled (CodeRabbit Round 3 deferred #5)

- [ ] `web/landing/index.html:610-612` text reading "deploy serverless v0.2.0" is reconciled with the surrounding "Release v0.3.0-alpha" heading.
- [ ] Decision rule: if the runtime serverless image was actually upgraded to v0.3.0-alpha as part of the v0.3.0-alpha release, update the runtime number; otherwise rename the heading to match runtime, OR explicitly distinguish "Release v0.3.0-alpha (gateway)" from "Runtime v0.2.0 (serverless)". @dev verifies which is correct against `serverless/Dockerfile` / actual deploy state and chooses the most accurate copy.

### AC5 — Hero feature cards mention text generation

- [ ] Hero section feature cards at `web/landing/index.html` (currently three: Async API, API Key Auth, T2I + I2I) are updated to acknowledge the text-gen modality. Two acceptable approaches:
  - Extend "T2I + I2I" → "T2I + I2I + Text" (single card, minimal churn), OR
  - Add a fourth card "Text Generation" / "Gemma 4" if the grid layout permits without breaking responsive breakpoints.
- [ ] @dev picks the approach that fits cleanest into the existing Tailwind grid; AC5 is satisfied either way.

### AC6 — Image-gen content untouched

- [ ] After commit, `git diff origin/main...HEAD -- web/landing/index.html` shows ONLY the additions described in AC1, AC2, AC5 plus the surgical fixes in AC3 and AC4. No reflow, no removed lines from existing image-gen sections.
- [ ] Existing image-gen sections (T2I block, I2I block, errors list except the AC3 addition, all `/jobs` references, version cards except the AC4 copy fix) are byte-identical to their pre-story state.

### AC7 — Browser smoke

- [ ] Page renders without console errors when opened in a modern browser (Chrome / Firefox latest).
- [ ] Anchor link `#endpoints` correctly scrolls to API Reference section and the new `/v1/generate` block is visible.
- [ ] Anchor link `#quickstart` correctly scrolls to Quickstart and both image-gen and text-gen sub-sections are visible.
- [ ] Anchor link `#errors` correctly scrolls to Errors and the new `invalid_image_base64` entry is visible.
