## Acceptance Criteria

> Derivados de `docs/prd/epic-2-consumer-integration.md` v0.4 вЂ” seГ§ГЈo "Preliminary ACs вЂ” Story 2.2 вЂ” TypeScript SDK". Ancorados em ADR-0001 Impact + PM Addendum 1.

1. **Typed client contract** вЂ” Exportar `FluxClient` class com construtor `new FluxClient({ apiKey: string, gatewayUrl: string, options?: ClientOptions })` e mГ©todos tipados: `warmup()`, `generate(input)`, `isWarm()`, `getLastWarmTimestamp()`. Input de `generate()` com types estritos (`steps: number`, `width: number`, `height: number`, `prompt: string`, `seed?: number`) вЂ” rejeita `{steps: "4"}` ou `{width: 0}` com `ValidationError` **antes** de hit no gateway (zero coerГ§ГЈo implГ­cita).

2. **Warmup helper** вЂ” `warmup(options?: { timeout?: number }): Promise<WarmupResult>` dispara dummy minimal request (`{ prompt: "warmup", steps: 1, width: 512, height: 512 }`) e resolve quando worker estГЎ hot. Default timeout **180s** (honra PRD v0.3 SLA cold ceiling). Retorna `{ duration_ms: number, was_cold: boolean }`. HeurГ­stica `was_cold = duration_ms > 30_000`. Side-effect: atualiza `lastWarmTimestamp` interno.

3. **Retry-with-backoff** вЂ” `generate()` com retry config default: `{ maxRetries: 3, backoffStrategy: 'exponential', initialDelayMs: 1000 }` (delays 1sв†’2sв†’4s). First-attempt timeout 180s (cold SLA); subsequent 30s (warm assumption). Configurable via constructor options. Se todos retries falharem por cold timeout в†’ throw `ColdStartError`. Se 429 в†’ throw `RateLimitError` imediato (sem retry em rate limit).

4. **Typed error classes** вЂ” MГіdulo `errors.ts` expondo (todas extend `Error`, todas testГЎveis via `instanceof` em CJS e ESM):
   - `ColdStartError`: props `{ duration_ms, retry_count, last_http_status? }`
   - `RateLimitError`: props `{ retry_after_seconds, reset_at }` (parsed de `Retry-After` header + body)
   - `AuthError`: props `{ http_status: 401 }`
   - `ValidationError`: props `{ field, reason }` (lanГ§ada pre-network em AC1)
   - `NetworkError`: props `{ cause: Error }` (wrap de erros de fetch)

5. **is-warm probe** вЂ” `isWarm(): Promise<boolean>` lightweight: se `lastWarmTimestamp` < 30s atrГЎs в†’ return `true` (cached, zero network). Se > 30s ou `null` в†’ return `false` (conservative). Expor `getLastWarmTimestamp(): number | null`. NГѓO conta contra retry budget e NГѓO emite request de network (pure state check).

6. **Gateway integration** вЂ” SDK chama **exclusivamente** `gatewayUrl` da Story 2.5, nunca RunPod direto. Todas requests: `X-API-Key: ${apiKey}` header + `Content-Type: application/json`. Parse de response 429 lГЄ `Retry-After` header + body `{error, limit, reset_at}` в†’ `RateLimitError`. Parse de 401 в†’ `AuthError`. `RUNPOD_API_KEY` **zero referГЄncias** no cГіdigo SDK (lives sГі no gateway Worker secret).

7. **Documentation** вЂ” `README.md` da package com quickstart executГЎvel (<30 linhas cГіdigo):
   - InstalaГ§ГЈo (`npm install @gemma4/flux-client` + `.npmrc` config)
   - Init client (`new FluxClient({ apiKey, gatewayUrl })`)
   - Warmup on app init (`await client.warmup()`)
   - Generate com try/catch tipado (`ColdStartError`, `RateLimitError`)
   - Retry config override example
   - CГіdigo copy-paste deve funcionar contra gateway real (validaГ§ГЈo: copiar em projeto test, `npm install`, `npx tsx quickstart.ts` в†’ success).

8. **Package publishing** вЂ” npm pacote `@gemma4/flux-client` publicado em **GitHub Packages** (privado, escopo `@gemma4`). Build output: dual CJS + ESM + types (.d.ts). VersГЈo inicial `0.1.0` (semver). `package.json` exige:
   - `"main"` (CJS entry), `"module"` (ESM entry), `"types"` (d.ts), `"exports"` field completo
   - `"engines": { "node": ">=18" }` (native fetch requirement)
   - ValidaГ§ГЈo: `npm pack` gera tarball; `npm install @gemma4/flux-client` em projeto test resolve types (autocomplete funciona em VSCode).
