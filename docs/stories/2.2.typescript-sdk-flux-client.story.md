## Story

**As a** consumidor TypeScript (web demo da Story 2.3, aplicações futuras npm-install-ready),
**I want** um SDK `@gemma4/flux-client` que encapsula chamadas ao gateway (Story 2.5) tratando cold-start realism (~130s) via `warmup()` + retry-with-backoff, com types estritos para input e error classes tipadas (`ColdStartError`, `RateLimitError`, `ValidationError`),
**so that** o código consumidor não precisa reimplementar retry/warmup/timeout handling e pode confiar em `catch (e instanceof ColdStartError)` para UX flows sem conhecer detalhes de RunPod ou rate-limit do gateway.

## Acceptance Criteria

> Derivados de `docs/prd/epic-2-consumer-integration.md` v0.4 — seção "Preliminary ACs — Story 2.2 — TypeScript SDK". Ancorados em ADR-0001 Impact + PM Addendum 1.

1. **Typed client contract** — Exportar `FluxClient` class com construtor `new FluxClient({ apiKey: string, gatewayUrl: string, options?: ClientOptions })` e métodos tipados: `warmup()`, `generate(input)`, `isWarm()`, `getLastWarmTimestamp()`. Input de `generate()` com types estritos (`steps: number`, `width: number`, `height: number`, `prompt: string`, `seed?: number`) — rejeita `{steps: "4"}` ou `{width: 0}` com `ValidationError` **antes** de hit no gateway (zero coerção implícita).

2. **Warmup helper** — `warmup(options?: { timeout?: number }): Promise<WarmupResult>` dispara dummy minimal request (`{ prompt: "warmup", steps: 1, width: 512, height: 512 }`) e resolve quando worker está hot. Default timeout **180s** (honra PRD v0.3 SLA cold ceiling). Retorna `{ duration_ms: number, was_cold: boolean }`. Heurística `was_cold = duration_ms > 30_000`. Side-effect: atualiza `lastWarmTimestamp` interno.

3. **Retry-with-backoff** — `generate()` com retry config default: `{ maxRetries: 3, backoffStrategy: 'exponential', initialDelayMs: 1000 }` (delays 1s→2s→4s). First-attempt timeout 180s (cold SLA); subsequent 30s (warm assumption). Configurable via constructor options. Se todos retries falharem por cold timeout → throw `ColdStartError`. Se 429 → throw `RateLimitError` imediato (sem retry em rate limit).

4. **Typed error classes** — Módulo `errors.ts` expondo (todas extend `Error`, todas testáveis via `instanceof` em CJS e ESM):
   - `ColdStartError`: props `{ duration_ms, retry_count, last_http_status? }`
   - `RateLimitError`: props `{ retry_after_seconds, reset_at }` (parsed de `Retry-After` header + body)
   - `AuthError`: props `{ http_status: 401 }`
   - `ValidationError`: props `{ field, reason }` (lançada pre-network em AC1)
   - `NetworkError`: props `{ cause: Error }` (wrap de erros de fetch)

5. **is-warm probe** — `isWarm(): Promise<boolean>` lightweight: se `lastWarmTimestamp` < 30s atrás → return `true` (cached, zero network). Se > 30s ou `null` → return `false` (conservative). Expor `getLastWarmTimestamp(): number | null`. NÃO conta contra retry budget e NÃO emite request de network (pure state check).

6. **Gateway integration** — SDK chama **exclusivamente** `gatewayUrl` da Story 2.5, nunca RunPod direto. Todas requests: `X-API-Key: ${apiKey}` header + `Content-Type: application/json`. Parse de response 429 lê `Retry-After` header + body `{error, limit, reset_at}` → `RateLimitError`. Parse de 401 → `AuthError`. `RUNPOD_API_KEY` **zero referências** no código SDK (lives só no gateway Worker secret).

7. **Documentation** — `README.md` da package com quickstart executável (<30 linhas código):
   - Instalação (`npm install @gemma4/flux-client` + `.npmrc` config)
   - Init client (`new FluxClient({ apiKey, gatewayUrl })`)
   - Warmup on app init (`await client.warmup()`)
   - Generate com try/catch tipado (`ColdStartError`, `RateLimitError`)
   - Retry config override example
   - Código copy-paste deve funcionar contra gateway real (validação: copiar em projeto test, `npm install`, `npx tsx quickstart.ts` → success).

8. **Package publishing** — npm pacote `@gemma4/flux-client` publicado em **GitHub Packages** (privado, escopo `@gemma4`). Build output: dual CJS + ESM + types (.d.ts). Versão inicial `0.1.0` (semver). `package.json` exige:
   - `"main"` (CJS entry), `"module"` (ESM entry), `"types"` (d.ts), `"exports"` field completo
   - `"engines": { "node": ">=18" }` (native fetch requirement)
   - Validação: `npm pack` gera tarball; `npm install @gemma4/flux-client` em projeto test resolve types (autocomplete funciona em VSCode).

### SDK Design Reference (REUSE > ADAPT > CREATE)

- **REUSE candidates** (study for patterns, not copy):
  - `replicate` npm package — retry + error class pattern
  - `@ai-sdk/openai` (Vercel) — dual bundle + types approach
  - `openai` official Node SDK — warmup não tem, mas retry/timeout architecture similar
- **ADAPT** — retry-with-backoff pattern é commoditizado; adaptar de um dos acima
- **CREATE** — `warmup()` helper + cold-specific timeout logic é único (nossa constraint do ADR-0001 Path A)

### Error Hierarchy Design

```
Error
 ├── ColdStartError       (cold timeout após retries)
 ├── RateLimitError       (429, sem retry automático)
 ├── AuthError            (401, sem retry)
 ├── ValidationError      (pre-network, input inválido)
 └── NetworkError         (DNS/connection, wrap de fetch error)
```

Consumers **devem** fazer `catch (e instanceof ColdStartError)` para diferentes UX flows — story 2.3 faz isso explicitamente.
