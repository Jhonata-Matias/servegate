> **Rationale:** this is a documentation UX/content-architecture story. The work is primarily MDX content restructuring and Astro Starlight sidebar configuration. It must not change gateway, SDK, serverless, or model-runtime behavior.

## Story

**As a** developer evaluating servegate's public documentation,
**I want** the docs portal organized around human tasks and a clear learning journey,
**so that** I can understand what servegate does, generate or edit images, generate text, check job status, and handle errors without decoding raw endpoint order first.

## Acceptance Criteria

1. **Sidebar follows the target reader journey.**
   - The Starlight sidebar in `web/docs/astro.config.mjs` is reorganized into this order:
     1. Welcome
     2. Quickstart
     3. Capabilities
        - Generate images (`generate-images`)
        - Edit images (`edit-images`)
        - Generate text (`generate-text`)
     4. Reference
        - API Reference
        - SDK (TypeScript)
        - Handle errors
     5. Reports
        - RunPod Video Capacity Report
   - The existing `RunPod Video Report` is moved out of the top of the sidebar and into `Reports`.
   - The three capability entries are implemented as standalone MDX pages under `web/docs/src/content/docs/` to keep the learning path discoverable from sidebar and search.

2. **Capability content teaches before reference.**
   - The docs introduce `Generate images`, `Edit images`, `Generate text`, and `Check job status` in plain language before showing raw endpoint names, JSON payloads, or error tables.
   - Each capability introduction answers:
     - What this does
     - When to use it
     - Where to go next

3. **API Reference is reorganized by capability.**
   - `web/docs/src/content/docs/api.mdx` follows this internal order:
     1. Overview
     2. Generate images
     3. Edit images
     4. Check job status
     5. Generate text
     6. Rate limits and headers
     7. Endpoint-specific error notes
   - `Generate text` appears before any dedicated error taxonomy section.
   - Raw endpoint names such as `POST /jobs`, `GET /jobs/{job_id}`, and `POST /v1/generate` are used as supporting labels or subtitles, not as the only top-level H2 when a clearer user action exists.

4. **Repeated API section pattern is applied.**
   - Each major API capability section uses the pattern:
     - When to use this
     - Endpoint
     - Request
     - Response
     - Common mistakes
   - If a field does not apply to a capability, the section may explicitly say so instead of omitting context silently.

5. **Errors page becomes recovery-focused.**
   - The sidebar label is changed from `Errors` to `Handle errors`.
   - `web/docs/src/content/docs/errors.mdx` begins with practical recovery guidance before exhaustive tables.
   - Full error taxonomy remains on this page; `api.mdx` keeps only endpoint-specific error notes and links to `/errors/`.

6. **Welcome and Quickstart cards use human-friendly labels.**
   - `web/docs/src/content/docs/index.mdx` links to the updated learning path using task-oriented card titles/descriptions.
   - `web/docs/src/content/docs/quickstart.mdx` no longer frames text generation as a "bonus"; it is presented as a first-class documented capability or clearly linked as the next capability after image jobs.

7. **Terminology is introduced in plain language.**
   - First use of T2I, i2i, SSE, SDK, RunPod, and Pagefind in the touched pages includes a short plain-language explanation or avoids the acronym in headings.
   - Main headings prefer action-oriented labels such as `Generate images`, `Edit images`, `Check job status`, `Generate text`, and `Handle errors`.

8. **Scope guard is preserved.**
   - No files under `gateway/`, `sdk/`, `serverless/`, or `spike/` are modified.
   - No runtime contract, API behavior, SDK behavior, or deployment target is changed.
   - The dark Starlight theme is preserved; no visual redesign or new custom component is required.

9. **Validation passes.**
   - `pnpm typecheck` passes from `web/docs`.
   - `pnpm build` passes from `web/docs`.
   - Generated docs navigation renders all new or renamed pages without broken internal links.
