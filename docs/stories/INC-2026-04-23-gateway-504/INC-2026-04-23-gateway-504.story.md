## Story

**As a** direct API consumer of the servegate gateway (Postman / curl / third-party integrators / SDK),
**I want** POST `/` substituído por contrato async submit/poll (POST `/jobs` + GET `/jobs/{id}`) com o SDK `@jhonata-matias/flux-client` bumped para v0.2.0 usando esse padrão internamente,
**so that** jobs em cold-start (~130s per ADR-0001 Path A) completem com sucesso em vez de falhar com HTTP 504 no timeout cliff de 60s do Cloudflare Worker, E eu possa testar o gateway via Postman/curl sem depender do warmup logic do SDK que hoje mascara o problema apenas para seus consumidores.

## Acceptance Criteria

> Derivados de `requirements.functional` + `requirements.nonFunctional` (validados em Phase 1 e corrigidos em Phase 3 research). 8 ACs totais cobrindo os 6 FRs ativos + 3 NFRs com validação empírica.

1. **POST /jobs async contract (FR-1, NFR-1).** POST `/jobs` com `{prompt, steps?, seed?, width?, height?}` + `X-API-Key` retorna HTTP `202` em **<2s p95** (Worker `wallTimeMs < 2000`), body `{job_id, status_url, est_wait_seconds: "unknown"}`, headers `Location: /jobs/{job_id}` e `Retry-After: 5` per RFC 7231. `job_id` é UUID v4 gerado via `crypto.randomUUID()`.

2. **GET /jobs/{id} poll contract (FR-2).** GET `/jobs/{id}` cobre os 4 estados terminais/transitórios:
   - `200` + `{output: {image_b64, metadata}}` quando RunPod `/status` retorna `status=COMPLETED`
   - `202` + `{status: "queued"|"running", est_wait_seconds: "unknown"}` quando em progresso
   - `504` + `{error: "generation_timeout", timeout_s: 280}` quando `TIMED_OUT`
   - `404` + `{error: "job_not_found_or_expired"}` para `job_id` inexistente/expirado (unified — não revela qual)

3. **KV mapping + TTL corretos (FR-3).** Mapping `job_id → runpod_request_id` persistido em namespace `JOBS_KV` com TTL **completion+30min** (alinhado com RunPod async retention de 30min) OU **submit+6h** se nunca completar. `updateStatus` em `COMPLETED` re-escreve entry com novo TTL.

4. **RunPod env var hot-fix (FR-4).** `COMFY_GENERATION_TIMEOUT_S=280` verificado em produção RunPod (dashboard → Serverless → servegate endpoint → Environment Variables). Evidence capturada em `.aiox/notes/FR-4-runpod-env-audit-2026-04-23.md` (screenshot ou captura textual).

5. **SDK v0.2.0 publicado com taxonomy corrigida (FR-6 + AD-2).** `@jhonata-matias/flux-client@0.2.0` publicado com `⚠️ BREAKING` em CHANGELOG. `generate(input)` preserva signature pública mas internamente submete via POST `/jobs` + polls GET `/jobs/{id}` respeitando `Retry-After`. `ColdStartError` **removido**; `TimeoutError` com campo `cause: 'poll_exhausted' | 'gateway_504' | 'runpod_timeout'` adicionado. Migration note explícita no CHANGELOG.

6. **est_wait_seconds sempre "unknown" (FR-7).** Todas as respostas 202 (POST `/jobs` e GET `/jobs/{id}`) contêm literal `est_wait_seconds: "unknown"` — nenhum número, nenhuma heurística (simplificação MVP per OQ-5).

7. **Zero 504 cold-start em produção (NFR-3).** Após 24h de observação pós-deploy em produção: `Count(status == 504 AND wallTimeMs >= 60000) == 0` verificado via Cloudflare Analytics. Qualquer ocorrência triggers rollback e iteração.

8. **POST / removido (CON-6, EC-8).** POST na rota raiz retorna **404** (ou 405) com body `{error: "endpoint_removed", migration_doc: "/docs/api/migration-async.md"}`. Zero fallback/503 temporário — deprecação atômica per user decision em OQ-1.

### 🔑 Architectural Decisions (MUST honor during implementation)

Estas decisões foram tomadas em Phase 6 (Plan) após Phase 5 (Critique) e resolvem 2 dos 4 MEDIUM issues do critique. **Não re-abra estas discussões durante dev** — elas já foram avaliadas com evidência.

#### AD-1 — Gateway `runpod.ts` NÃO chama `/view` (resolves CRIT-1)

**Decisão:** `runpod.ts` expõe apenas `submitJob()` → `/run` e `getStatus()` → `/status`. Zero chamadas a `/view` do gateway.

**Rationale:**
- `serverless/handler.py:237` retorna `{"image_b64": image_b64, "metadata": {...}}`
- RunPod Serverless framework marshala o retorno do handler no campo `output` do `/v2/{endpoint}/status/{id}` response
- Portanto quando `status=COMPLETED`, `output.image_b64` já contém os bytes em base64 — gateway NÃO precisa fetch adicional
- `/view` é endpoint INTERNO do worker (ComfyUI → handler.py) e não está exposto no API `/v2/` da RunPod
- Verificado via inspeção direta do handler.py existente + research RT-2 (`docs.runpod.io/serverless/endpoints/operation-reference`)

**Trade-off considerado e rejeitado:** design defensivo (probe inline primeiro, fallback para /view) — rejeitado porque adiciona complexidade sem benefício; handler.py é controlado por nós e `CON-3` locka esse contrato inalterado.

**Implicação para dev:** se em dev time você descobrir que `/status.output` retorna `null` ou um pointer em vez de `image_b64` direto, **pare e escale** — isso contradiz esta decisão e precisa re-análise do @architect. Não improvise um `/view` fallback.

#### AD-2 — SDK v0.2.0 REMOVE `ColdStartError`, ADD `TimeoutError` (resolves CRIT-4)

**Decisão:**
- `sdk/src/errors.ts`:
  - **REMOVE** `ColdStartError` export (deprecated em v0.2.0)
  - **ADD** `TimeoutError` (new class extends Error) com campo `cause: 'poll_exhausted' | 'gateway_504' | 'runpod_timeout'` e opcional `elapsedMs`
  - **PRESERVE** `RateLimitError`, `AuthError`, `NetworkError`, `ValidationError` (unchanged)

**Rationale:**
- Na arquitetura async, cold-start **não é um erro** — é uma fase de polling que o SDK absorve transparentemente
- O único terminal failure que faz sentido expor é "timeout": SDK esgotou tentativas de poll, OU gateway retornou 504 (RunPod TIMED_OUT)
- Repurposing semântico de `ColdStartError` seria enganoso — o nome implica cold, agora significaria timeout
- `ASM-4` confirma zero third-party consumers → breaking change é custo zero externamente

**Migration obrigatória em CHANGELOG v0.2.0:**

```markdown
## [0.2.0] — 2026-04-23

### ⚠️ BREAKING

- **Removed** `ColdStartError`. Cold-start is no longer a terminal error; SDK now
  transparently polls the async job endpoint.
- **Added** `TimeoutError` with `cause` field (`'poll_exhausted' | 'gateway_504' | 'runpod_timeout'`).

  **Migration:**
  ```ts
  // v0.1.0
  try { await client.generate(input) }
  catch (e) { if (e instanceof ColdStartError) { /* retry or alert */ } }

  // v0.2.0
  try { await client.generate(input) }
  catch (e) { if (e instanceof TimeoutError) { /* retry or alert */ } }
  ```

### Changed

- `generate()` now internally submits via POST /jobs + polls GET /jobs/{id}. Public signature unchanged.
- Internal timeouts (180s/30s in v0.1.0) replaced by poll loop bounded by `ClientOptions.warmTimeoutMs` (default: 180s).
```
