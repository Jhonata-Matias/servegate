## Story

## Acceptance Criteria

1. **pt-BR files created (4)** — `README.pt-BR.md` (raiz), `docs/usage/dev-onboarding.pt-BR.md`, `docs/legal/TERMS.pt-BR.md`, `docs/legal/PRIVACY.pt-BR.md`. Padrão de nome: `<nome>.pt-BR.md` ao lado do arquivo EN canônico. Encoding UTF-8 sem BOM, line endings LF.

2. **Language switcher banner** — cada um dos 4 arquivos EN + 4 pt-BR (8 arquivos total) recebe banner no topo (logo após H1 título, antes de qualquer outro conteúdo), formato padrão:
   - No EN: `> 🌐 **English** | [Português (Brasil)](./<nome>.pt-BR.md)`
   - No pt-BR: `> 🌐 [English](./<nome>.md) | **Português (Brasil)**`
   - Link relativo com `./` para funcionar tanto em GitHub quanto em cópia local/raw.
   - README raiz aponta para `./README.pt-BR.md` (sem prefixo `docs/`).

3. **Anglicismos técnicos preservados** — termos de vocabulário dev cotidiano NÃO são traduzidos: `API key`, `endpoint`, `rate limit`, `cold start`, `deploy`, `request`, `response`, `gateway`, `SDK`, `npm install`, `GitHub`, `header`, `token`, `Pull Request`, `issue`. Comandos, valores, hashes e nomes próprios preservados literalmente (ex: `npm install @jhonata-matias/flux-client`, commit `fd1232b`, `X-API-Key`, `100 requests/dia`). Prosa conectiva em pt-BR natural (artigos, verbos, preposições).

4. **Legal bilingual equivalence clause** — `TERMS.md`, `TERMS.pt-BR.md`, `PRIVACY.md`, `PRIVACY.pt-BR.md` recebem nova seção final "Bilingual Equivalence / Equivalência Bilíngue" com texto espelhado:
   - **EN (TERMS):** "Both the English and Portuguese (Brazil) versions of these Terms are canonical and equally binding. In case of divergence, the version corresponding to the user's country of residence prevails."
   - **pt-BR (TERMS):** "Ambas as versões em Inglês e Português (Brasil) destes Termos são canônicas e igualmente vinculantes. Em caso de divergência, prevalece a versão correspondente ao país de domicílio do usuário."
   - **EN (PRIVACY):** mesma cláusula substituindo "these Terms" por "this Privacy Statement".
   - **pt-BR (PRIVACY):** mesma cláusula substituindo "destes Termos" por "desta Declaração de Privacidade".

5. **Structural parity** — cada arquivo `.pt-BR.md` mantém estrutura idêntica ao EN canônico: mesmos headings (H1-H6, mesma ordem + mesmo nível), mesmas listas/tabelas com mesma quantidade de colunas e linhas, mesmas URLs externas (não traduzidas), mesmos code blocks intactos (comandos, JSON, JS/TS, YAML). Nenhuma seção adicionada ou omitida — exceto a cláusula de AC4 que aparece em ambas versões dos legais.

6. **Cross-linking integrity** — links internos em `.pt-BR.md` apontam preferencialmente para a versão pt-BR correspondente se ela existir no escopo desta story (ex: `README.pt-BR.md` → `docs/usage/dev-onboarding.pt-BR.md` → `docs/legal/TERMS.pt-BR.md` → `docs/legal/PRIVACY.pt-BR.md`). Links para arquivos que NÃO foram traduzidos nesta story (`docs/api/reference.md`, `docs/usage/gateway-deploy.md`, `docs/usage/monitoring.md`, `docs/usage/comfyui-flux-quickstart.md`, `docs/usage/runpod-serverless-deploy.md`, `sdk/README.md`, `examples/colab/README.md`) apontam para versão EN acrescidos de marca `(em inglês)` adjacente ao texto do link.

7. **Quality review (standalone BR smoke)** — dev (ou `@qa`) executa o onboarding seguindo APENAS `dev-onboarding.pt-BR.md` **sem consultar a versão EN** em momento algum. Smoke inclui: (a) leitura completa dos 4 arquivos pt-BR em GitHub preview, (b) simular abrir issue via template `access-request.yml` (não precisa submeter), (c) ler TERMS.pt-BR + PRIVACY.pt-BR end-to-end, (d) simular setup de `.npmrc` e `npm install` (não precisa chave real). Resultado documentado em `docs/qa/2.7-translation-review.md` com: checklist de clareza por arquivo, typos/gramática encontrados+corrigidos na mesma PR, gaps descobertos (não-bloqueantes viram follow-up issue, bloqueantes voltam para `@dev` fix antes de close).

### Strategic Decisions (from Epic 2 PRD v0.9)

Três decisões tomadas com user em 2026-04-22 que definem este scope:

1. **Escopo MVP = 4 arquivos da jornada crítica** — não traduz API reference, usage secundários, SDK README nesta story. Se demanda BR confirmar pós-MVP → Epic 3 backlog.
2. **Estrutura = sufixo `.pt-BR.md` + banner bilíngue** — padrão usado por Vue, Astro, Tailwind. Não quebra links existentes em nenhum doc que referencia o nome EN (há ~23 refs no repo, inspecionadas no rename gemma4→servegate).
3. **Canonical = ambos bilíngue (legal)** via cláusula de equivalência prevalecendo versão do domicílio do usuário. Alinha com CDC art. 46-54 (direito a informação clara em pt-BR para consumidor BR) sem duplicar ônus de manutenção legal em dois idiomas source-of-truth. Estilo: **anglicismos técnicos preservados** (dev BR real fala "API key", não "chave de API").

**Manter em EN (anglicismos técnicos cotidianos do dev):**

| Termo EN | NÃO traduzir para |

| API key | ~~chave de API~~ |

| rate limit / rate-limit | ~~limite de taxa~~, ~~limite de requisições~~ |
| cold start | ~~partida a frio~~ |
| deploy | ~~implantação~~ |
| request | ~~requisição~~ (exceto em contexto CDC não-técnico) |
| response | ~~resposta de API~~ |
| gateway | ~~porta de entrada~~ |
| SDK | ~~Kit de Desenvolvimento~~ |

| Contexto EN | Tradução pt-BR natural |

| "Once you have a key..." | "Quando você tiver uma key..." |

| "Before you start" | "Antes de começar" |
| "If you don't agree" | "Se você não concorda" |
| "For each request..." | "Para cada request..." |
